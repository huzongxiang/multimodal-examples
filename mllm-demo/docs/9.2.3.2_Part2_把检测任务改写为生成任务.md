# 9.2.3.2 Part 2：把检测任务改写为生成任务——从自然语言描述走向结构化输出

## 1. 阶段目标

Part 2 的任务是将目标检测改写为文本生成问题，使模型从“根据图像生成自然语言描述”进一步过渡到“根据图像生成结构化结果”。这一阶段的关键不在于追求专门检测器意义上的最优定位精度，而在于回答另一个更重要的问题：在已经建立起视觉—语言连接之后，统一的生成式接口能否承担结构化视觉任务。

从阶段链路上看，Part 2 直接继承 Part 1 的主分支权重。换言之，模型不再从单独的视觉编码器和语言模型裸底座重新开始，而是在已有的跨模态连接基础上继续训练。这样设计的目的，是检验 Part 1 所建立的视觉前缀机制是否足以支撑更受约束、更强调格式稳定性的输出任务。

因此，Part 2 的重点并不是“检测精度是否超过专门检测网络”，而是三个方面：第一，模型是否能够稳定输出可解析的 JSON；第二，模型是否能够在文本生成过程中保留目标类别信息；第三，模型是否能够用生成文本近似表达图像中的局部空间位置。

## 2. 数据与任务形式

这一阶段使用动物检测数据集，共 `800` 条训练样本，并保留 `200` 条测试样本用于评估。与 Part 1 不同的是，目标文本不再是自然语言句子，而是显式的 JSON 风格结果。每条记录包含图像路径、一组候选提示模板以及一个目标 JSON 字符串，目标 JSON 中写入了对象类别和归一化边界框。

样本记录的组织形式如下。

```json
{
  "image": "data/repo/part2_animals_od/train/images/00000.jpg",
  "task": "object_detection",
  "prompt_options": [
    "What objects are in this image? Output as JSON.",
    "Detect all objects in this image and return JSON with bounding boxes.",
    "List the objects with their locations in JSON format.",
    "Find all objects and output their bounding boxes as JSON."
  ],
  "target": "{\"objects\": [{\"label\": \"cat\", \"bbox\": [0.003, 0.1, 0.726, 0.864]}]}",
  "append_eos": true,
  "mask_prompt_loss": true
}
```

这一数据格式体现了与 Part 1 明显不同的任务约束。首先，目标检测阶段引入了显式提示词，模型不再是在空提示条件下生成任意描述，而是在问题导向的文本上下文中输出结果。其次，训练目标不再是开放式句子，而是结构化 JSON。再次，`mask_prompt_loss` 被设置为 `true`，说明提示词部分只作为条件输入，不参与损失计算，模型真正需要学习的是提示词之后的结构化目标文本。

从模型配置上看，Part 2 继续使用 `google/vit-large-patch16-224` 作为视觉编码器，使用 `HuggingFaceTB/SmolLM-360M` 作为语言模型，并从 `output/part1` 中加载上一阶段权重作为初始化。训练使用 `batch_size=4`、`lr=1e-4`、`max_length=256`，共训练 `49` 个 epoch，总步数为 `9800`，并在第 `49` 个 epoch 触发早停。

## 3. 训练与推理主线

Part 2 在模型主体上与 Part 1 保持一致，变化集中在数据格式、提示模板和初始化来源上。阶段训练仍然由统一入口 `train.py` 管理，只是在命令行中显式指定上一阶段权重与更长的文本长度。训练命令如下。

```powershell
$env:PYTHONPATH = "src"
python -m mllm_demo.train `
  --train-jsonl "data\repo\part2_animals_od\train\data.jsonl" `
  --output-dir "output\part2" `
  --checkpoint-path "output\part2\training_checkpoint.pt" `
  --init-from "output\part1" `
  --vision-model "google/vit-large-patch16-224" `
  --lm-model "HuggingFaceTB/SmolLM-360M" `
  --epochs 50 `
  --batch-size 4 `
  --lr 1e-4 `
  --max-length 256 `
  --early-stop-patience 5 `
  --early-stop-min-delta 0.003
```

这一阶段的训练逻辑相对于 Part 1 有两点关键变化。第一，输入文本从“空提示 + 图像描述”变为“检测提示 + JSON 目标”，模型在训练中需要显式区分条件提示和真正预测目标。第二，目标文本的正确性不再只是语义合理，而是同时要求格式合法、类别正确、位置表达近似合理。这样一来，模型要学习的不仅是“说一句话”，而是“按指定格式给出结构化答案”。

推理阶段仍然沿用统一的生成接口，只是提示词被固定为检测任务相关模板。例如，在单样本推理中，可以使用如下命令：

```powershell
$env:PYTHONPATH = "src"
python -m mllm_demo.infer `
  --checkpoint-dir "output\part2" `
  --image-path "data\repo\part2_animals_od\test\images\00022.jpg" `
  --prompt "What objects are in this image? Output as JSON." `
  --max-new-tokens 128
```

模型输出之后，还需要进一步将文本解析为 JSON，并从中提取对象类别和边界框。这一点与 Part 1 有本质区别：Part 1 的结果可以直接按自然语言解释，而 Part 2 的结果只有在结构保持稳定时才真正有意义。因此，推理阶段的关键在于“生成结果是否可解析、可使用”。

## 4. 结果分析

从训练过程看，Part 2 的收敛要比 Part 1 更稳定。该阶段最佳 epoch 为 `44`，最佳 epoch 平均损失约为 `0.0046`，最终 epoch 平均损失约为 `0.0061`，并在第 `49` 个 epoch 触发早停。这说明在更强约束的结构化任务下，模型比自由图像描述任务更容易获得稳定的优化方向。

从小样本评估结果看，Part 2 是整个阶段链路中表现最稳定的单任务阶段之一。对 `12` 条测试样本进行评估后，模型在检测任务上的 `json_parse_success` 为 `1.0`，`object_count_accuracy` 约为 `0.9167`，`label_accuracy` 约为 `0.7917`，平均 `bbox_iou` 约为 `0.4078`。这些结果表明，模型已经能够稳定输出合法 JSON，类别恢复能力较好，空间位置也具备可用的近似表达能力。

从样例输出看，模型在这一阶段的典型行为具有明显的阶段特征。对于单目标图像，模型通常能够给出正确类别，并生成一个大致合理的边界框；对于双目标图像，类别往往仍然能够保持正确，但目标数量有时会低估。例如，在一张包含两只鸡的图像中，模型只输出了一个 `chicken` 对象，导致目标数准确率下降；在 racoon 和 goat 等样本中，类别预测正确，框位置虽然不够精确，但仍然与真实目标具有显著重合。这说明模型已经掌握了“按结构输出目标信息”的基本能力，但在多目标场景中的局部细节还不够稳定。

为了更具体地说明这一阶段的输入输出关系，下面给出两个代表性成功样例。

样例一体现了单目标场景下的稳定表现：

- 输入图像：`test/images/00022.jpg`
- 提示词：`What objects are in this image? Output as JSON.`
- 参考输出：`{"objects": [{"label": "racoon", "bbox": [0.163, 0.023, 0.801, 0.661]}]}`
- 模型输出：`{"objects": [{"label": "racoon", "bbox": [0.309, 0.07, 0.539, 0.816]}]}`

在该样例中，模型正确恢复了目标类别，目标数量也保持正确，边界框虽然与参考框存在差异，但仍然覆盖了主要目标区域，单样本 IoU 约为 `0.5184`。

样例二体现了双目标场景下的类别保持能力：

- 输入图像：`test/images/00006.jpg`
- 提示词：`What objects are in this image? Output as JSON.`
- 参考输出：`{"objects": [{"label": "cow", "bbox": [0.447, 0.017, 0.462, 0.82]}, {"label": "cow", "bbox": [0.306, 0.284, 0.312, 0.529]}]}`
- 模型输出：`{"objects": [{"label": "cow", "bbox": [0.563, 0.196, 0.362, 0.706]}, {"label": "cow", "bbox": [0.045, 0.183, 0.295, 0.706]}]}`

在该样例中，模型保持了合法 JSON 结构，两个对象的类别也都恢复正确，只是边界框位置仍然比较粗，单样本平均 IoU 约为 `0.2941`。这说明模型已经具备多目标结构化输出能力，但定位精度仍有明显提升空间。

该阶段的意义已经十分清楚。首先，它说明目标检测可以被改写为文本生成问题，而且这种改写不仅停留在概念上，而是能够产出机器可解析、类别大体正确、位置近似可用的结果。其次，它说明 Part 1 建立的视觉前缀机制已经不再只会生成长段自然语言，而是可以在更强格式约束下输出结构化信息。再次，它说明统一的生成式建模方式不仅适用于图像描述，也能够延伸到传统上通常依赖专门任务头的视觉任务。

同时，Part 2 的边界也很明确。平均 IoU 仍然明显低于专门检测网络所追求的定位精度，多目标场景中的计数稳定性也不够充分，个别样本中还会出现目标数低估或类别判断错误。因此，这一阶段更适合被理解为“结构化视觉任务生成化已经成立”，而不是“文本生成方式已经替代专门检测器”。对本章而言，Part 2 最重要的结论不是检测性能的绝对数值，而是统一生成接口已经从自然语言描述成功扩展到了结构化视觉预测。

因此，Part 2 在整个实验链路中的作用可以概括为：模型开始从“描述图像”转向“表达图像结构”，统一生成接口已经能够承担受格式约束的视觉任务。正是在这一基础上，系统才有条件继续进入下一阶段，即在图像与语言双条件下完成视觉问答任务。
