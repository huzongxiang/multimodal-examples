# 9.1.3 分阶段实验与结果分析

## 1、Part 1：最小视觉语言模型的建立

**1）阶段目标**

Part 1 的任务是建立最小视觉语言模型的基本工作链路，使图像能够进入语言模型上下文，并驱动文本生成。该阶段的关键不在于获得高质量图像描述性能，而在于回答一个更基础的问题：当视觉编码器输出的连续特征经过映射后，是否能够作为语言模型的条件前缀参与下一个 token 预测。

因此，Part 1 需要完成三项工作。第一，建立视觉编码器、映射层和语言模型之间的连接路径。第二，使用最基本的图像描述任务验证这一连接路径可以被训练。第三，为后续结构化检测和视觉问答阶段提供可继续优化的初始化权重。沿着这一目标，Part 1 在整个实验链路中承担的是跨模态连接起点的作用。

**2）数据与任务形式**

这一阶段使用 Flickr8k 子集，共 `2000` 条样本。每条记录包含一张图像和五条候选描述，训练时从候选描述中随机选择一条作为当前目标文本。任务形式非常直接：输入为图像，输出为一条自然语言描述。由于该阶段没有额外提示词，因此训练目标就是最基本的图像条件下的文本生成。

样本记录的组织形式如下。

```json
{
  "image": "data/repo/part1_flickr8k/images/00000.jpg",
  "task": "caption",
  "captions": [
    "Boys with their backs against an incoming wave.",
    "Four boys are about to be hit by an approaching wave.",
    "Four people sitting in the path of a wave.",
    "Four young men are sitting on the beach under a crashing wave.",
    "three boys sitting in sand getting splashed by wave"
  ],
  "append_eos": false,
  "mask_prompt_loss": false
}
```

这一数据格式有两个特点。其一，图像描述阶段没有显式提示词，模型直接根据图像前缀进入生成。其二，由于目标文本本身就是完整预测对象，因此 `mask_prompt_loss` 为 `false`，即文本序列中的全部有效 token 都参与损失计算。这种设置使 Part 1 成为整个系统中最纯粹的“图像到文本”生成阶段。

从模型配置上看，Part 1 使用 `google/vit-large-patch16-224` 作为视觉编码器，使用 `HuggingFaceTB/SmolLM-360M` 作为语言模型，视觉分支保持冻结，仅训练映射层与语言模型参数。训练使用 `batch_size=4`、`lr=2e-4`、`max_length=64`，共训练 `50` 个 epoch，总步数为 `25000`。

**3）训练与推理主线**

Part 1 虽然任务简单，但它完整经过了从数据读取、图像编码、文本生成到权重保存的全部训练过程。阶段训练仍然由统一入口 `train.py` 负责，差别主要体现在本阶段使用的模型规格、样本组织方式和超参数上。训练命令如下。

```powershell
$env:PYTHONPATH = "src"
python -m mllm_demo.train `
  --train-jsonl "data\repo\part1_flickr8k\train.jsonl" `
  --output-dir "output\part1" `
  --checkpoint-path "output\part1\training_checkpoint.pt" `
  --vision-model "google/vit-large-patch16-224" `
  --lm-model "HuggingFaceTB/SmolLM-360M" `
  --epochs 50 `
  --batch-size 4 `
  --lr 2e-4 `
  --max-length 64
```

这一阶段的训练逻辑可以概括为以下四步。

1. 数据模块读取图像和候选描述，随机选取当前目标句子，并构造文本 token 与标签。
2. 模型将图像送入视觉编码器，得到 patch 级视觉特征，再通过 `projector` 投影到语言模型隐藏空间。
3. 图像前缀与文本嵌入拼接为统一上下文，语言模型在该上下文上执行因果语言建模，并计算下一个 token 预测损失。
4. 训练模块按 step 和 epoch 记录损失，并在阶段结果中保留 checkpoint、最终权重、损失曲线和运行摘要。

在这一过程中，图像并未通过单独任务头直接映射到描述结果，而是被当作语言模型序列前缀的一部分。模型学习到的不是“先看图，再调用一个描述器”，而是“在图像前缀存在时，语言模型如何延续出一段合理文本”。这正是后续阶段继续沿用统一生成接口的前提。

推理阶段沿用同一结构。模型先读取保存好的 `MiniVLM` 权重，对输入图像进行编码，然后直接执行自回归生成。由于该阶段没有必须给定的提示词，输入可以为空。单样本推理命令如下。

```powershell
$env:PYTHONPATH = "src"
python -m mllm_demo.infer `
  --checkpoint-dir "output\part1" `
  --image-path "data\repo\part1_flickr8k\images\00000.jpg" `
  --max-new-tokens 64
```

由此，Part 1 已经形成了一条完整的图像描述链路：图像输入、视觉编码、映射投影、文本生成以及阶段权重保存全部贯通。这一链路是否成立，将直接决定后续检测和问答阶段能否继续建立在同一骨架之上。

**4）结果分析**

从训练结果看，Part 1 已经形成稳定的跨模态连接路径。该阶段最终完成 `50` 个 epoch 训练，最佳 epoch 为 `49`，最佳 epoch 平均损失约为 `0.5566`，最终 epoch 平均损失约为 `0.5616`。从损失曲线看，模型在训练过程中表现出持续下降趋势，说明“图像前缀 + 文本生成”这一训练路径是可优化的，视觉特征确实能够进入语言模型并影响生成行为。

但是，如果把 Part 1 直接当作一个独立的图像描述系统来评价，其结果并不理想。对 `12` 条图像描述样本进行小样本评估后，图像描述任务的 `exact match` 为 `0.0`，`token F1` 约为 `0.1284`。从生成样例看，模型虽然能够输出语法完整的自然语言段落，但语义偏移明显，且经常出现冗长、重复或发展为与图像内容无关的叙述。例如，在面对购物车场景图像时，模型输出了一段关于普通人与宗教主题的长文本；在面对节庆人群图像时，模型生成了关于聚会与吉他演奏的叙述；在若干样本中，输出甚至会演化为反复重复的主题短语。这些现象说明模型已经学会“生成语言”，但尚未稳定学会“根据图像生成正确描述”。

这一结果意味着，Part 1 的意义主要体现在结构层面，而不是性能层面。它已经证明以下几点成立：

1. 视觉特征经过映射后可以被送入语言模型上下文。
2. 统一的因果语言建模目标能够用于训练这一跨模态连接。
3. 阶段输出权重可以作为后续 Part 2 和 Part 3 的初始化来源。

与此同时，Part 1 也显示出明显边界。仅依靠小规模图像描述数据和最小连接方式，模型尚不足以形成稳定的高质量图像描述能力。换言之，该阶段更适合被理解为“最小跨模态连接的建立”，而不是“图像描述问题的充分解决”。正因为如此，后续阶段才有必要引入更受约束的结构化检测任务和条件问答任务，进一步推动模型从“能够生成文本”走向“能够按照任务要求生成正确文本”。

因此，Part 1 在整个实验链路中的结论可以概括为：最小视觉语言连接已经建立，生成式训练路径已经打通，但系统能力仍主要停留在跨模态对齐起点上。后续阶段的目标，不再只是让模型继续生成语言，而是让它逐步学会按任务格式生成语言、按条件问题生成答案，并最终在统一接口下切换不同任务行为。

## 2、Part 2：把检测任务改写为生成任务

**1）阶段目标**

Part 2 的任务是将目标检测改写为文本生成问题，使模型从“根据图像生成自然语言描述”进一步过渡到“根据图像生成结构化结果”。这一阶段的关键不在于追求专门检测器意义上的最优定位精度，而在于回答另一个更重要的问题：在已经建立起视觉—语言连接之后，统一的生成式接口能否承担结构化视觉任务。

从阶段链路上看，Part 2 直接继承 Part 1 的主分支权重。换言之，模型不再从单独的视觉编码器和语言模型裸底座重新开始，而是在已有的跨模态连接基础上继续训练。这样设计的目的，是检验 Part 1 所建立的视觉前缀机制是否足以支撑更受约束、更强调格式稳定性的输出任务。

因此，Part 2 的重点并不是“检测精度是否超过专门检测网络”，而是三个方面：第一，模型是否能够稳定输出可解析的 JSON；第二，模型是否能够在文本生成过程中保留目标类别信息；第三，模型是否能够用生成文本近似表达图像中的局部空间位置。

**2）数据与任务形式**

这一阶段使用动物检测数据集，共 `800` 条训练样本，并保留 `200` 条测试样本用于评估。与 Part 1 不同的是，目标文本不再是自然语言句子，而是显式的 JSON 风格结果。每条记录包含图像、一组候选提示模板以及一个目标 JSON 字符串，目标 JSON 中写入了对象类别和归一化边界框。

样本记录的组织形式如下。

```json
{
  "image": "data/repo/part2_animals_od/train/images/00000.jpg",
  "task": "object_detection",
  "prompt_options": [
    "What objects are in this image? Output as JSON.",
    "Detect all objects in this image and return JSON with bounding boxes.",
    "List the objects with their locations in JSON format.",
    "Find all objects and output their bounding boxes as JSON."
  ],
  "target": "{\"objects\": [{\"label\": \"cat\", \"bbox\": [0.003, 0.1, 0.726, 0.864]}]}",
  "append_eos": true,
  "mask_prompt_loss": true
}
```

这一数据格式体现了与 Part 1 明显不同的任务约束。首先，目标检测阶段引入了显式提示词，模型不再是在空提示条件下生成任意描述，而是在问题导向的文本上下文中输出结果。其次，训练目标不再是开放式句子，而是结构化 JSON。再次，`mask_prompt_loss` 被设置为 `true`，说明提示词部分只作为条件输入，不参与损失计算，模型真正需要学习的是提示词之后的结构化目标文本。

从模型配置上看，Part 2 继续使用 `google/vit-large-patch16-224` 作为视觉编码器，使用 `HuggingFaceTB/SmolLM-360M` 作为语言模型，并以前一阶段权重作为初始化。训练使用 `batch_size=4`、`lr=1e-4`、`max_length=256`，共训练 `49` 个 epoch，总步数为 `9800`，并在第 `49` 个 epoch 触发早停。

**3）训练与推理主线**

Part 2 在模型主体上与 Part 1 保持一致，变化集中在数据格式、提示模板和初始化来源上。阶段训练仍然由统一入口 `train.py` 管理，只是在命令行中显式指定上一阶段权重与更长的文本长度。训练命令如下。

```powershell
$env:PYTHONPATH = "src"
python -m mllm_demo.train `
  --train-jsonl "data\repo\part2_animals_od\train\data.jsonl" `
  --output-dir "output\part2" `
  --checkpoint-path "output\part2\training_checkpoint.pt" `
  --init-from "output\part1" `
  --vision-model "google/vit-large-patch16-224" `
  --lm-model "HuggingFaceTB/SmolLM-360M" `
  --epochs 50 `
  --batch-size 4 `
  --lr 1e-4 `
  --max-length 256 `
  --early-stop-patience 5 `
  --early-stop-min-delta 0.003
```

这一阶段的训练逻辑相对于 Part 1 有两点关键变化。第一，输入文本从“空提示 + 图像描述”变为“检测提示 + JSON 目标”，模型在训练中需要显式区分条件提示和真正预测目标。第二，目标文本的正确性不再只是语义合理，而是同时要求格式合法、类别正确、位置表达近似合理。这样一来，模型要学习的不仅是“说一句话”，而是“按指定格式给出结构化答案”。

推理阶段仍然沿用统一的生成接口，只是提示词被固定为检测任务相关模板。例如，在单样本推理中，可以使用如下命令：

```powershell
$env:PYTHONPATH = "src"
python -m mllm_demo.infer `
  --checkpoint-dir "output\part2" `
  --image-path "data\repo\part2_animals_od\test\images\00022.jpg" `
  --prompt "What objects are in this image? Output as JSON." `
  --max-new-tokens 128
```

模型输出之后，还需要进一步将文本解析为 JSON，并从中提取对象类别和边界框。这一点与 Part 1 有本质区别：Part 1 的结果可以直接按自然语言解释，而 Part 2 的结果只有在结构保持稳定时才真正有意义。因此，推理阶段的关键在于“生成结果是否可解析、可使用”。

**4）结果分析**

从训练过程看，Part 2 的收敛要比 Part 1 更稳定。该阶段最佳 epoch 为 `44`，最佳 epoch 平均损失约为 `0.0046`，最终 epoch 平均损失约为 `0.0061`，并在第 `49` 个 epoch 触发早停。这说明在更强约束的结构化任务下，模型比自由图像描述任务更容易获得稳定的优化方向。

从小样本评估结果看，Part 2 是整个阶段链路中表现最稳定的单任务阶段之一。对 `12` 条测试样本进行评估后，模型在检测任务上的 `json_parse_success` 为 `1.0`，`object_count_accuracy` 约为 `0.9167`，`label_accuracy` 约为 `0.7917`，平均 `bbox_iou` 约为 `0.4078`。这些结果表明，模型已经能够稳定输出合法 JSON，类别恢复能力较好，空间位置也具备可用的近似表达能力。

从样例输出看，模型在这一阶段的典型行为具有明显的阶段特征。对于单目标图像，模型通常能够给出正确类别，并生成一个大致合理的边界框；对于双目标图像，类别往往仍然能够保持正确，但目标数量有时会低估。例如，在一张包含两只鸡的图像中，模型只输出了一个 `chicken` 对象，导致目标数准确率下降；在 racoon 和 goat 等样本中，类别预测正确，框位置虽然不够精确，但仍然与真实目标具有显著重合。这说明模型已经掌握了“按结构输出目标信息”的基本能力，但在多目标场景中的局部细节还不够稳定。

为了更具体地说明这一阶段的输入输出关系，下面给出两个代表性成功样例。

1. 样例一体现了单目标场景下的稳定表现：

- 输入图像：`test/images/00022.jpg`
- 提示词：`What objects are in this image? Output as JSON.`
- 参考输出：`{"objects": [{"label": "racoon", "bbox": [0.163, 0.023, 0.801, 0.661]}]}`
- 模型输出：`{"objects": [{"label": "racoon", "bbox": [0.309, 0.07, 0.539, 0.816]}]}`

在该样例中，模型正确恢复了目标类别，目标数量也保持正确，边界框虽然与参考框存在差异，但仍然覆盖了主要目标区域，单样本 IoU 约为 `0.5184`。

2. 样例二体现了双目标场景下的类别保持能力：

- 输入图像：`test/images/00006.jpg`
- 提示词：`What objects are in this image? Output as JSON.`
- 参考输出：`{"objects": [{"label": "cow", "bbox": [0.447, 0.017, 0.462, 0.82]}, {"label": "cow", "bbox": [0.306, 0.284, 0.312, 0.529]}]}`
- 模型输出：`{"objects": [{"label": "cow", "bbox": [0.563, 0.196, 0.362, 0.706]}, {"label": "cow", "bbox": [0.045, 0.183, 0.295, 0.706]}]}`

在该样例中，模型保持了合法 JSON 结构，两个对象的类别也都恢复正确，只是边界框位置仍然比较粗，单样本平均 IoU 约为 `0.2941`。这说明模型已经具备多目标结构化输出能力，但定位精度仍有明显提升空间。

该阶段的意义已经十分清楚。首先，它说明目标检测可以被改写为文本生成问题，而且这种改写不仅停留在概念上，而是能够产出机器可解析、类别大体正确、位置近似可用的结果。其次，它说明 Part 1 建立的视觉前缀机制已经不再只会生成长段自然语言，而是可以在更强格式约束下输出结构化信息。再次，它说明统一的生成式建模方式不仅适用于图像描述，也能够延伸到传统上通常依赖专门任务头的视觉任务。

同时，Part 2 的边界也很明确。平均 IoU 仍然明显低于专门检测网络所追求的定位精度，多目标场景中的计数稳定性也不够充分，个别样本中还会出现目标数低估或类别判断错误。因此，这一阶段更适合被理解为“结构化视觉任务生成化已经成立”，而不是“文本生成方式已经替代专门检测器”。对本章而言，Part 2 最重要的结论不是检测性能的绝对数值，而是统一生成接口已经从自然语言描述成功扩展到了结构化视觉预测。

因此，Part 2 在整个实验链路中的作用可以概括为：模型开始从“描述图像”转向“表达图像结构”，统一生成接口已经能够承担受格式约束的视觉任务。正是在这一基础上，系统才有条件继续进入下一阶段，即在图像与语言双条件下完成视觉问答任务。

## 3、Part 3：视觉问答

**1）阶段目标**

Part 3 的任务是在已经建立视觉前缀机制和结构化生成能力之后，引入图像与文本问题的双条件输入，使模型能够根据图像内容回答外部问题。该阶段的关键不在于把系统变成一个单独的分类器，而在于检验统一的生成式接口是否能够从“图像驱动文本生成”进一步推进到“图像与语言条件共同驱动答案生成”。

与 Part 1 相比，Part 3 不再要求模型自由描述图像；与 Part 2 相比，Part 3 也不再主要依赖结构化格式约束，而是要求模型根据问题理解图像中的关键信息，并生成一个与问题匹配的短答案。因此，这一阶段检验的重点不再是结构保持，而是视觉理解与语言条件之间的结合是否已经成立。

**2）数据与任务形式**

这一阶段使用视觉问答数据，共 `3600` 条训练样本与 `400` 条评估样本。每条记录包含图像、问题提示词和目标答案。任务形式统一写成 `Question: ... Answer:`，模型需要在图像前缀和文本问题的共同条件下生成答案。

样本记录的组织形式如下。

```json
{
  "image": "data/repo/part3_vqav2/images/00000.jpg",
  "task": "vqa",
  "prompt": "Question: Where is he looking? Answer:",
  "target": "down",
  "append_eos": true,
  "mask_prompt_loss": true
}
```

这一数据格式与前两个阶段相比有两个关键变化。第一，提示词中显式包含一个自然语言问题，模型必须把问题本身纳入生成条件。第二，目标文本通常很短，往往只有一个词或短语，因此这一阶段对“图像与语言条件是否匹配”的要求高于对长文本生成流畅性的要求。

从模型配置上看，Part 3 继续使用 `google/vit-large-patch16-224` 作为视觉编码器，使用 `HuggingFaceTB/SmolLM-360M` 作为语言模型，并以前一阶段权重作为初始化。训练使用 `batch_size=8`、`lr=2e-4`、`max_length=64`，共训练 `14` 个 epoch，总步数为 `5400`，并在第 `14` 个 epoch 触发早停。

**3）训练与推理主线**

Part 3 在模型主体上仍然与前两个阶段一致，变化集中在输入文本组织方式和目标类型上。训练命令如下。

```powershell
$env:PYTHONPATH = "src"
python -m mllm_demo.train `
  --train-jsonl "data\repo\part3_vqav2\train.jsonl" `
  --output-dir "output\part3" `
  --checkpoint-path "output\part3\training_checkpoint.pt" `
  --init-from "output\part1" `
  --vision-model "google/vit-large-patch16-224" `
  --lm-model "HuggingFaceTB/SmolLM-360M" `
  --epochs 50 `
  --batch-size 8 `
  --lr 2e-4 `
  --max-length 64 `
  --early-stop-patience 5 `
  --early-stop-min-delta 0.005
```

这一阶段的训练逻辑可以概括为三点。第一，图像仍然通过视觉编码器和映射层进入语言模型上下文，模型主体保持不变。第二，提示词中的问题文本成为显式条件，模型必须在图像前缀基础上进一步理解“问题在问什么”。第三，输出答案通常较短，这使得模型是否真正抓住图像中的关键信息，比是否能够维持长文本连贯性更重要。

推理阶段同样沿用统一接口，只是提示词改为具体问题。例如，可以使用如下命令进行单样本推理：

```powershell
$env:PYTHONPATH = "src"
python -m mllm_demo.infer `
  --checkpoint-dir "output\part3" `
  --image-path "data\repo\part3_vqav2\images\03879.jpg" `
  --prompt "Question: Is the dog chasing the sheep? Answer:" `
  --max-new-tokens 16
```

与 Part 2 不同，Part 3 的结果通常无需再进行 JSON 解析，关键在于生成的答案是否与问题语义和图像内容一致。因此，推理阶段的判断标准不再是“输出是否结构合法”，而是“输出是否答对了问题”。

**4）结果分析**

从训练结果看，Part 3 的损失下降很快。该阶段最佳 epoch 为 `9`，最佳监控值约为 `0.00433`，最终 epoch 平均损失约为 `0.00116`，并在第 `14` 个 epoch 触发早停。这说明在相对短答案的问答任务上，模型能够较快适应新的目标格式。

从小样本评估结果看，Part 3 已经表现出基础视觉问答能力。对 `12` 条评估样本进行测试后，`exact match` 为 `0.3333`，`token F1` 同样为 `0.3333`。这一结果说明模型能够在部分样本上给出正确答案，但整体准确率仍然有限，尚不足以支撑稳定的高质量视觉问答能力。

从具体样例看，这一阶段已经出现了多种能够答对的问题类型。

1. 样例一体现了关系判断上的正确回答：

- 输入图像：`images/03879.jpg`
- 提示词：`Question: Is the dog chasing the sheep? Answer:`
- 参考答案：`yes`
- 模型输出：`yes`

该样例说明模型已经能够把图像中的对象关系与问题条件对应起来，并生成正确的 yes/no 答案。

2. 样例二体现了人数计数上的正确回答：

- 输入图像：`images/03814.jpg`
- 提示词：`Question: How many people are there? Answer:`
- 参考答案：`2`
- 模型输出：`2`

该样例说明模型在部分计数问题上已经能够给出与图像一致的短答案。

3. 样例三体现了场景属性判断上的正确回答：

- 输入图像：`images/03829.jpg`
- 提示词：`Question: Is that a chain shower curtain? Answer:`
- 参考答案：`no`
- 模型输出：`no`

该样例说明模型不仅能回答数量问题，也能够在二元判断类问题中保持较稳定的输出。

这一阶段同样存在明显边界。以树木计数问题为例，模型在如下样例中给出了错误答案：

- 输入图像：`images/03887.jpg`
- 提示词：`Question: How many trees? Answer:`
- 参考答案：`6`
- 模型输出：`3`

这一现象说明模型虽然已经形成了条件回答能力，但在细粒度计数和属性判断上仍然不够稳定。

由这些结果可以看出，Part 3 的阶段意义已经非常明确。首先，模型已经从“图像到文本”推进到“图像 + 文本条件到答案”的生成形式，说明统一接口能够支持更明确的指令条件。其次，模型开始表现出基础问答能力，这意味着视觉前缀机制不再只服务于图像描述或结构化输出，而开始服务于条件性回答行为。再次，这一阶段也暴露出比检测任务更高的难度：在缺少强格式约束的情况下，模型更容易产生语义上合理但事实错误的答案。

因此，Part 3 更适合被理解为“条件生成能力已经形成，但问答精度仍有限”。它并没有把视觉问答问题充分解决，却清楚地说明了同一多模态骨架已经能够支持图像与语言双条件生成。这一结论为下一阶段的多任务统一奠定了基础，因为只有当图像描述、目标检测和视觉问答都能够在同一接口上成立时，多任务混合训练才真正具有意义。

## 4、Part 4：多任务统一

**1）阶段目标**

Part 4 的任务是将图像描述、目标检测和视觉问答三类任务统一到同一个模型中，使系统在共享参数的条件下，能够根据不同提示词在不同任务之间切换行为。该阶段的重点不再是证明单一任务能否成立，而是检验统一生成接口是否已经足以支撑多任务系统的基本形态。

与前面三个阶段相比，Part 4 的核心变化不在模型结构，而在训练数据的组织方式和模型分支的选择。该阶段使用更小的一组底座配置，即 `ViT-Base + SmolLM-135M`，并在正式多任务训练前先进行一个图像描述预热步骤，以获得适合该分支的初始化权重。因此，Part 4 观察的重点不再只是单任务效果，而是统一接口下的任务切换能力。

**2）数据与任务形式**

这一阶段使用 `2400` 条混合样本，其中图像描述、目标检测和视觉问答样本分别为 `1000`、`400` 和 `1000` 条。三类样本被统一组织到同一个训练集合中，模型仍然只接收图像和文本序列，不同任务之间的区分主要依赖提示模板与目标文本格式。

多任务样本的组织可以分为三类。

图像描述样本示例：

```json
{
  "image": "data/repo/part4_multitask/images/caption/00000.jpg",
  "task": "caption",
  "prompt": "Describe this image.",
  "captions": [
    "Boys with their backs against an incoming wave .",
    "Four boys are about to be hit by an approaching wave .",
    "Four people sitting in the path of a wave ."
  ],
  "append_eos": true,
  "mask_prompt_loss": true
}
```

目标检测样本与 Part 2 类似，仍以 JSON 目标为主；视觉问答样本则继续使用 `Question: ... Answer:` 模板。这样一来，多任务训练中的关键变化就不再是数据格式是否统一，而是三类任务是否能够在同一训练过程中共享同一套模型主体。

从模型配置上看，Part 4 使用 `google/vit-base-patch16-224` 作为视觉编码器，使用 `HuggingFaceTB/SmolLM-135M` 作为语言模型，并以预热阶段权重作为初始化。训练使用 `batch_size=4`、`lr=1e-4`、`max_length=256`，共训练 `50` 个 epoch，总步数为 `30000`。

**3）训练与推理主线**

Part 4 的训练仍然由统一入口 `train.py` 完成，只是在输入数据和初始化权重上发生变化。训练命令如下。

```powershell
$env:PYTHONPATH = "src"
python -m mllm_demo.train `
  --train-jsonl "data\repo\part4_multitask\train.jsonl" `
  --output-dir "output\part4" `
  --checkpoint-path "output\part4\training_checkpoint.pt" `
  --init-from "output\part1_for_part4" `
  --vision-model "google/vit-base-patch16-224" `
  --lm-model "HuggingFaceTB/SmolLM-135M" `
  --epochs 50 `
  --batch-size 4 `
  --lr 1e-4 `
  --max-length 256 `
  --early-stop-patience 5 `
  --early-stop-min-delta 0.005
```

这一阶段的训练重点不再是让模型适应某一类单独目标文本，而是让同一模型在交替出现的三类任务记录上学会不同的输出模式。图像描述样本要求生成自然语言描述，目标检测样本要求生成结构化 JSON，视觉问答样本要求生成短答案文本。模型主体、损失函数和前向路径全部保持不变，真正承担任务区分作用的是提示模板和目标格式。

推理阶段同样沿用统一接口，只需替换提示词即可切换任务。例如：

- 图像描述：`Describe this image.`
- 目标检测：`What objects are in this image? Output as JSON.`
- 视觉问答：`Question: What is being celebrated? Answer:`

这种推理方式意味着，多任务系统的“多任务”并不是通过三套不同推理函数实现的，而是通过同一生成入口在不同提示词条件下表现出的不同行为实现的。

**4）结果分析**

从训练结果看，Part 4 完整训练了 `50` 个 epoch，最佳 epoch 为 `47`，最佳 epoch 平均损失约为 `0.3336`，最终 epoch 平均损失约为 `0.3365`。从过程上看，这一阶段已经形成稳定的多任务训练权重，并未在设定轮数内触发早停。

从平衡小样本评估结果看，Part 4 是整个实验链路中最能体现统一接口能力的阶段。对 `18` 条平衡样本进行评估后，三类任务的结果如下。

- 图像描述：`exact match = 0.1667`，`token F1 ≈ 0.7964`
- 目标检测：`json_parse_success = 1.0`，`object_count_accuracy ≈ 0.8333`，`label_accuracy = 1.0`，`bbox_iou ≈ 0.8595`
- 视觉问答：`exact match = 1.0`，`token F1 = 1.0`

需要注意的是，这一评估集来自训练混合集合中的平衡抽样，因此它更适合说明“统一接口下的多任务行为已经形成”，而不宜直接理解为严格的泛化性能结论。

为了更具体地说明这一阶段的输入输出关系，下面分别给出三类任务的样例。

1. 图像描述样例

- 输入图像：`images/caption/00777.jpg`
- 提示词：`Describe this image.`
- 参考描述之一：`Two children are riding horses over sand near a white fence .`
- 模型输出：`Two children are riding horses over sand near a white fence .`

该样例说明，在多任务训练条件下，模型已经能够在统一提示词引导下恢复与图像内容高度一致的自然语言描述。

2. 目标检测样例

- 输入图像：`images/od/00172.jpg`
- 提示词：`What objects are in this image? Output as JSON.`
- 参考输出：`{"objects": [{"label": "person", "bbox": [0.211, 0.402, 0.406, 0.402]}, {"label": "dog", "bbox": [0.514, 0.428, 0.433, 0.381]}]}`
- 模型输出：`{"objects": [{"label": "person", "bbox": [0.211, 0.402, 0.406, 0.402]}, {"label": "dog", "bbox": [0.519, 0.383, 0.381, 0.301]}]}`

该样例说明模型不仅保持了合法 JSON 结构，而且正确恢复了对象类别，边界框也与参考结果较为接近，单样本 IoU 约为 `0.7678`。

3. 视觉问答样例

- 输入图像：`images/vqa/00094.jpg`
- 提示词：`Question: What is being celebrated? Answer:`
- 参考答案：`birthday`
- 模型输出：`birthday`

该样例说明，在统一模型参数下，系统不仅能够描述图像和输出结构化检测结果，还能够在问答模板下切换为短答案生成行为。

4. 多任务切换现象

除上述成功样例之外，Part 4 的输出还显示出典型的任务切换现象。例如，在目标检测任务中，模型可以直接输出 `racoon`、`goat`、`dog` 等对象的 JSON；在视觉问答任务中，又能够对 `What toppings are on the pizza?` 生成 `cheese and oregano`，对 `Is this an unhealthy snack?` 生成 `no`。这说明模型并不是简单记住某一种输出风格，而是在提示词的驱动下切换不同任务模式。

该阶段的意义因此十分明确。首先，它说明统一接口不仅适用于单任务，而是已经能够组织起图像描述、目标检测与视觉问答三类行为。其次，它说明最小多模态系统的关键特征，不仅在于能够同时处理图像和文本，还在于能够在同一套参数下复用同一推理入口完成不同任务。再次，它说明多任务训练在本实验链路中起到了真正的收束作用：前面各阶段形成的能力，在这一阶段被组织到同一个模型之中。

同时，Part 4 的边界也需要明确。由于当前平衡评估样本来自训练混合数据，结果更适合说明“任务切换能力和统一接口已经成立”，而不适合直接表述为严格的泛化验证。换言之，Part 4 的主要结论不是“已经得到一个高性能通用多模态模型”，而是“已经形成一个具有清晰任务切换能力的最小多任务多模态系统”。

因此，Part 4 在整个实验链路中的结论可以概括为：统一生成接口已经能够承载多种视觉语言任务，同一模型能够根据提示词在描述、检测和问答之间切换行为，最小 MLLM 的基本形态由此形成。
