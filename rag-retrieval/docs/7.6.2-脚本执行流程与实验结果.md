# 7.6.2 执行主链与结果验证

## 1. 脚本入口与运行方式

系统通过脚本 `run_pdf_qa.py` 执行索引、检索、问答与结果落盘。相较于界面入口，脚本更容易固定输入参数、重复运行流程并保留结果输出。

运行方式可以写为：

```bash
cd <project_root>
micromamba run -n <env_name> python run_pdf_qa.py
```

脚本默认执行以下动作：

1. 读取示例 PDF。
2. 索引前 `2` 页。
3. 使用内置问题集进行问答。
4. 将结果输出到 `<output_dir>`。

默认问题集如下：

```python
DEFAULT_QUESTIONS = [
    "What is the title of this supplementary material?",
    "Which conference submission is this supplementary material for?",
    "What datasets are listed in Table 4?",
    "What four metric panels are shown in Figure 10?",
]
```

如果仅需要验证本地页面级检索链，而不调用云端生成接口，则可使用：

```bash
micromamba run -n <env_name> python run_pdf_qa.py --retrieval-only
```

该模式将系统划分为两个可以独立验证的阶段：本地检索阶段与云端问答阶段。

## 2. 从索引到回答的执行闭环

从脚本实现看，整条主链可以归纳为四个步骤。

第一步，建立本次运行上下文。系统通过 `run_id` 隔离页面图像目录与本地 Milvus 数据文件：

```python
run_id = f"{pdf_path.stem}-{uuid.uuid4().hex[:8]}"
rag = None if args.retrieval_only else Rag()
middleware = Middleware(run_id, create_collection=True)
```

第二步，执行索引。系统把示例 PDF 转换为页面图像，并写入页面级向量存储：

```python
indexed_pages = middleware.index(
    pdf_path=str(pdf_path),
    id=run_id,
    max_pages=args.max_pages,
)
```

第三步，对每个问题执行页面级检索，并把命中的页面标识恢复为真实图像路径：

```python
search_result = middleware.search([question])[0]
top_score, top_doc_id = search_result[0]
page_num = top_doc_id + 1
image_path = BASE_DIR / "pages" / run_id / f"page_{page_num}.png"
```

第四步，在非 `retrieval_only` 模式下，将命中页图像交给多模态生成接口，并把结果统一写入 JSON 文件：

```python
results.append(
    {
        "question": question,
        "answer": answer,
        "page_num": int(page_num),
        "search_score": float(top_score),
        "image_path": str(image_path.resolve()),
        "glm_attempts": int(attempts),
    }
)
```

因此，该脚本并不只是“调用一次问答”，而是完成了索引、检索、回答与结果持久化的完整工程闭环。

## 3. 本地检索结果验证

在 `--retrieval-only` 模式下，系统不会调用云端生成模型，而只保留页面级检索结果。这一模式用于验证本地检索链本身是否成立。

以示例文档为例，三道代表性问题均稳定命中第 `1` 页，对应分数分别约为 `16.58`、`13.00` 与 `15.31`。这一结果说明：

1. 问题已经能够被映射到页面级多模态向量空间。
2. 本地 Milvus 检索链能够稳定返回相关页面。
3. 结果输出已经具备页号、得分与页面图像路径等最小复核信息。

对于工程系统而言，这一步非常关键。因为只有在检索链独立成立之后，后续多模态生成失败时，系统才能明确判断问题位于云端问答接口，而非本地索引与召回过程。

## 4. 全流程结果与案例分析

在完整问答模式下，系统对四个问题给出了如下结果：

| 问题类型 | 命中页 | 结果摘要 | 说明 |
| --- | --- | --- | --- |
| 标题识别 | 1 | 触发 `429` 限流 | 检索正确，生成阶段失败 |
| 会议投稿信息 | 1 | `CVPR 2025` | 文本类问题成功 |
| `Table 4` 数据集提取 | 1 | `MVTec AD, VisA, BTAD, DAGM, DTD-Synthetic` | 表格类问题成功 |
| `Figure 10` 面板识别 | 2 | `1-shot AUROC, 1-shot PRO, 4-shot AUROC, 4-shot PRO` | 图表类问题成功 |

第一，页面级检索已经有效成立。前三题命中第 `1` 页，图表题命中第 `2` 页，说明系统能够根据问题内容在不同页面之间定位证据来源。

第二，系统已经覆盖文本、表格与图表三类证据形态。它并非只能处理标题文本，而是已经能够在整页图像基础上消费表格与图表信息。

第三，系统当前的主要不稳定因素来自云端生成接口，而不是本地检索链。标题问题虽然未返回稳定答案，但命中页与检索分数均正常，失败原因是接口限流。

由此可见，当前系统已经不是单纯的页面检索演示样例，而是一个能够实际运行、能够产生可复核结果、同时清楚暴露工程约束的最小多模态问答原型。
