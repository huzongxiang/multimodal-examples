# 7.6.3 结论、边界与扩展方向

## 1. 案例已经验证的能力

前述系统搭建与结果验证已经表明，当前原型完成了三项关键证明。

第一，PDF 页面完全可以作为文档问答系统的第一检索对象。系统并未依赖 OCR 文本切块作为起点，而是直接从页面图像进入多模态 embedding 与页面级检索流程。

第二，页面级多模态检索已经足以支撑一个最小问答闭环。只要问题与证据集中分布在单页中，`ColPali + Milvus` 可以定位相关页，`GLM` 可以继续在命中页上完成阅读与回答。

第三，本地检索链与云端生成链可以被清晰分离。`retrieval_only` 结果已经证明本地阶段本身是成立的；完整问答结果则进一步说明，系统的不稳定性主要来自云端接口，而不是页面级索引与召回逻辑。

因此，当前系统的合理定位不是“完整文档推理系统”，而是一个**已经跑通页面级检索与页级问答闭环的多模态文档原型**。

## 2. 当前系统的能力边界

该原型之所以仍应被界定为原型，是因为它的边界同样清晰。

第一，证据粒度仍然停留在页面级。系统内部虽然利用 token 级向量进行匹配，但暴露给上层的结果仍是“哪一页最相关”，而不是“页内哪一块区域最相关”。

第二，生成接口仍然是整页输入。当前 `Rag` 发送给 `GLM` 的内容是“整页图像 + 问题文本”，系统尚未建立区域级裁剪、证据压缩和证据编排机制。

第三，答案与证据的绑定仍然较粗。结果文件可以说明答案来源于哪一页，但还不能稳定标注答案来自哪一个区域、哪一张表格、哪一条图例或哪一个对象。

第四，系统默认工作在 `topk=1` 的页面级场景下，因此更适合单页问答，而不适合跨页证据组织问题。若答案依赖多页共同支撑，现有链路会自然暴露上限。

这些边界并不削弱案例的示范价值，但它们决定了当前系统尚不能被等同于完整的文档智能平台。

## 3. 后续扩展的优先顺序

如果要在当前基础上继续推进，更关键的不是立即更换更大的模型，而是补足证据层次。

第一步，应扩展存储单元定义。当前 `Milvus` schema 主要保存 `vector`、`seq_id`、`doc_id` 和路径信息，后续应补充 `page_num`、`bbox`、`region_id`、`evidence_type`、`ocr_text` 等字段，使页面局部区域能够成为显式证据单元。

第二步，应在页面级召回之后增加区域级重排。现有 `Late Interaction` 逻辑已经提供了基础，后续需要变化的并不是核心打分公式，而是重排对象本身，应从“整页”推进为“区域块”或“对象块”。

第三步，应在 `Middleware` 与 `Rag` 之间加入证据组织层。当前系统检索后直接把整页交给生成模型，后续则应把命中结果组织为更低噪声的证据视图，例如“页面缩略图 + 局部截图 + OCR 片段 + 坐标与页码”。

第四步，应在输出层加入细粒度证据绑定机制。届时系统输出不应只有答案与页号，还应能够标注答案所依赖的页面、区域和对象来源。

由此可见，当前原型若继续演进，其主线并不是“换模型”，而是“补足证据粒度、证据组织和证据绑定”。只有在这三个层次逐步建立之后，系统才会从页面级原型进一步走向更完整的多模态文档系统。
