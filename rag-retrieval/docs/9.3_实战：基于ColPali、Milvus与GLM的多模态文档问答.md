# 9.3 实战：基于 ColPali、Milvus 与 GLM 的多模态文档问答

多模态文档问答的难点，不在于“文档里有图片”，而在于文档证据本身就是页面结构的一部分。标题与正文的相对位置、表格的列对齐关系、图表标题与图例的对应关系，经常和文本内容本身一样重要。若先把 PDF 压平为 OCR 文本切块，再进入普通文本检索流程，页面中的视觉结构就很容易被削弱；而一旦结构被削弱，表格、图表和复杂版式中的关键信息也会随之丢失。

这正是 `ColPali` 路线值得单独讨论的原因。`ColPali` 论文《ColPali: Efficient Document Retrieval with Vision Language Models》指出，文档页面可以直接由视觉语言模型编码为多向量表示，并结合 `Late Interaction` 完成页面级检索；Hugging Face 社区文章则进一步展示了这一思路如何与 `Milvus` 和视觉生成模型组合为可运行的多模态 RAG 主链。这样一来，检索阶段保留的就不只是词面内容，还包括表格布局、图表结构和版面关系所携带的视觉上下文。

当前项目沿用了这一主线，但把它收束为一个更适合当前案例分析的最小系统。系统先把 PDF 页面转成图像，再由 `ColPali` 生成页面与问题的多模态表示，本地 `Milvus` 负责完成页面级召回与重排，最后由 `GLM` 在命中页上执行阅读与回答。与社区文章中使用 `Gemini` 或 `GPT-4o` 作为回答端相比，这里把生成端替换为 `GLM`，从而更集中地观察页面级检索链与多模态回答链如何协同。

从实际运行结果看，这一原型已经形成了一个最小但完整的页面级问答闭环。系统索引了示例补充材料的前两页，并围绕标题识别、会议信息抽取、表格读取和图表面板识别四类问题执行检索与问答。结果显示：本地检索能够把标题、会议信息和表格问题稳定定位到第 `1` 页，把图表问题定位到第 `2` 页；在生成阶段，后三类问题均得到有效回答，只有标题问题因外部接口限流未返回稳定结果。这个现象非常重要，因为它表明当前案例真正验证的，不是“某个模型把答案都答对了”，而是“页面级多模态检索链能够成立，且与生成链之间的职责边界是清楚的”。
