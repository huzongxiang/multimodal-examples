# 9.3.3 结论、边界与扩展方向

在系统结构和运行结果都已展开之后，本节最后收束这一案例能够支持的判断、当前结论的适用边界，以及后续若继续推进时更值得优先补足的方向。换言之，这里关心的不是再重复前文事实，而是把“这个案例究竟证明了什么、没有证明什么、下一步应当往哪里扩展”说清楚。

## 1. 这个案例真正验证了什么

经过前两节的系统拆解和结果分析，可以把本案例的结论收束为三点。

第一，PDF 页面可以直接成为多模态检索对象，而不必先退化为 OCR 文本切块。页面级表示的价值，不只是省掉了一个文本抽取步骤，更在于它保留了标题、表格、图表和版面结构之间的关系。对于文档问答任务，这一点往往比单纯提取出一串文本更重要。

第二，`ColPali + Milvus` 已经足以承担一个最小页面级检索主链。ColPali 提供页面与问题的共享表示，Milvus 保留多向量并完成页面重排，这使系统能够在不做显式版面解析的前提下，把问题稳定路由到相关页面。当前样例里，文本、表格和图表三类问题都落在这条主链上，说明这一思路在当前案例中并不只适用于单一证据形态。

第三，本地检索链与多模态回答链之间的职责边界是清楚的。独立检索模式已经证明页面召回本身可以成立；全流程结果则说明，当回答端发生 `429` 限流时，检索端仍然能够给出正确页号。换言之，这个案例最有价值的地方，不是“答案全都答出来了”，而是“检索与回答已经可以分层分析”。

因此，当前系统的合理定位，不是完整的文档智能平台，而是一个**完成了页面级多模态检索与页级问答闭环验证的原型系统**。

## 2. 当前结论的边界在哪里

这个结论之所以成立，是因为它建立在一个边界非常明确的案例上；也正因如此，它的适用范围同样需要说清楚。

第一，当前验证规模仍然很小。案例只索引了一个补充材料样例的前两页，并围绕四类问题进行验证，其中完整问答真正成功回答的是三类问题。这样的结果足以证明方案可行，但还不足以支持更强的泛化判断。它说明的是“主链成立”，而不是“系统已经具备充分稳健性”。

第二，证据粒度仍停留在页面级。虽然 ColPali 内部使用的是 patch 级局部表示，Milvus 中保存的也不是单向量摘要，但系统最终返回给上层的仍然是“哪一页最相关”，而不是“页内哪一块区域最相关”。这意味着当前系统擅长的是页级定位，而不是细粒度区域引用。

第三，生成阶段仍然以整页为输入。当前回答模型接收的是“整页图像 + 问题文本”，因此它必须自行在整页中寻找相关区域。对于当前这种证据较集中、页数较少的案例，这样的做法仍然有效；但一旦页面密度提高、干扰信息增多，整页输入的噪声就会迅速放大。

第四，当前链路默认工作在 `topk=1` 的单页场景下。对于标题、会议信息、单页表格和单页图表，这一设定已经足够；但若问题需要跨页证据拼接，或者需要先在多个候选页之间比较，现有设计就会自然暴露上限。

第五，回答端稳定性仍依赖外部服务。本次运行中标题问题的失败并不是检索错误，而是外部接口限流。这说明在实际系统里，即便本地检索链已经稳定，整体体验仍然会受到回答端可用性、重试策略和服务配额的影响。

这些边界并不削弱案例的示范价值，但它们提醒我们：当前结果证明的是“页面级多模态文档问答可以成立”，而不是“完整文档问答系统已经构建完成”。

## 3. 从页面级原型走向更完整系统

如果要沿着这条路线继续推进，优先级最高的工作并不是立刻更换更大的模型，而是继续完善证据层次。

第一步，应把页面级召回进一步推进为区域级证据定位。当前检索已经具备 patch 级匹配基础，下一步更合理的方向不是推翻现有方案，而是把这些局部匹配结果显式提升为页内区域或对象级证据单元。只有这样，系统才会从“知道哪一页相关”走向“知道页内哪一块相关”。

第二步，应在页面召回与答案生成之间增加证据组织层。当前系统把整页直接交给回答模型，优点是链路短，缺点是噪声大。更成熟的方案应当在检索之后对证据进行整理，例如组合页面缩略图、局部裁剪和必要文字线索，从而让回答模型接收到更聚焦的输入。

第三步，应把单页问答扩展为多页证据组合。文档中的真实问题经常跨越多个页面，甚至需要把正文、表格和附图共同纳入推理。当前案例用 `topk=1` 验证了最小闭环，后续则应把“多页候选排序、证据拼接与答案生成”纳入统一流程。

第四步，应增加更明确的答案绑定机制。一个更成熟的系统不应只输出结论，还应指出答案依赖了哪一页、哪一块区域、哪一个表格或哪一个图表面板。只有当答案与证据之间形成显式绑定，这条链路才会从“能答”进一步走向“可解释、可核验”。

第五步，应补强回答端的工程稳健性。本次运行已经暴露出外部限流问题，因此后续如果要把系统推向更稳定的实用形态，还需要在回答端增加缓存、重试、降级策略和接口替代方案。

从这个角度看，当前原型最应该保留的并不是某个具体回答模型，而是“页面作为视觉检索对象、多向量页面重排、检索与生成明确分层”这一条方法主线。只要这一主线被保留下来，系统就可以在更细的证据粒度、更强的多页组织能力和更稳健的回答端支持下继续演进。
