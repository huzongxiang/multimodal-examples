# 9.3.3 结论、边界与扩展方向

这一案例的价值，主要体现在三个层面：它支持哪些判断，边界落在什么位置，以及后续沿着哪条路径继续推进最有效。

## 1. 这个案例验证了什么

经过前两节的系统拆解与结果分析，本案例的结论可以收束为三点。

第一，PDF 页面本身可以直接作为多模态检索对象。页面级表示保留了标题、表格、图表与版面结构之间的关系，这些信息本身就是文档问答的重要证据。对于复杂文档而言，这种页面级建模方式能够更完整地保存证据上下文。

第二，`ColPali + Milvus` 已经能够承担一个最小页面级检索主链。`ColPali` 提供页面与问题的共享表示，`Milvus` 保存多向量并完成页面重排，使系统可以在不做显式版面解析的条件下，把问题稳定路由到相关页面。当前样例中的文本、表格和图表问题都能沿着这条主链完成定位，说明这一思路已经覆盖了多种典型证据形态。

第三，检索链与回答链之间已经形成清晰分工。独立检索模式证明了页面召回本身可以稳定成立，全流程结果则进一步说明，检索端能够负责页号定位，回答端负责页内阅读与答案生成。即使标题问题受到 `429` 限流影响，检索端仍然保持了正确定位。由此，这个案例更准确的定位，是一个**完成了页面级多模态检索与页级问答闭环验证的原型系统**。

## 2. 当前结论的边界

这一案例同时给出了非常明确的适用边界。

第一，当前验证规模较小。案例围绕一个补充材料样例的前两页展开，并针对四类问题进行了验证，其中三类问题完成了稳定回答。这一结果已经足以说明主链可行，也为后续扩展提供了清楚起点。

第二，当前证据粒度停留在页面级。虽然 `ColPali` 内部使用 patch 级局部表示，`Milvus` 中保存的也不是单向量摘要，但系统向上层返回的结果仍然是“哪一页最相关”。因此，当前系统最擅长的是页级定位，下一步可以自然推进到页内区域级定位。

第三，回答阶段采用整页输入。当前回答模型接收“整页图像 + 问题文本”，因此适合处理页数较少、证据较集中的案例；当页面密度提高、干扰信息增多时，系统就更需要更细粒度的证据组织。

第四，当前链路工作在 `topk=1` 的单页场景下。对于标题、会议信息、单页表格和单页图表，这一设置已经能够很好支撑当前案例；若问题涉及跨页证据组合，系统就需要进入多页排序与证据拼接阶段。

第五，回答端稳定性仍然受外部服务条件影响。本次运行中的标题问题已经说明，回答链的整体体验与接口配额、限流控制和重试策略密切相关。因此，回答端工程稳健性也是系统继续推进时的重要组成部分。

这些边界共同说明，当前结果已经清楚证明了“页面级多模态文档问答主链可以成立”，并为下一阶段扩展划定了清晰范围。

## 3. 从页面级原型走向更完整系统

沿着这条路线继续推进，最值得优先补足的是证据层次与系统组织能力。

第一步，可以把页面级召回推进到区域级证据定位。当前检索已经具备 patch 级匹配基础，下一步最自然的方向，就是把这些局部匹配结果进一步提升为页内区域或对象级证据单元，使系统从“知道哪一页相关”发展到“知道页内哪一块相关”。

第二步，可以在页面召回与答案生成之间加入证据组织层。当前系统直接把整页交给回答模型，链路简洁、实现直接；进一步演进时，可以把页面缩略图、局部裁剪与必要文字线索重新组织成更聚焦的输入，让回答模型处理的信息更加集中。

第三步，可以把单页问答扩展为多页证据组合。当前案例已经验证了最小闭环，后续则可以把“多页候选排序、证据拼接与答案生成”整合进统一流程，使系统能够处理跨页问题。

第四步，可以补充更明确的答案绑定机制。系统在输出答案的同时，还可以同步给出对应页码、区域、表格或图表面板，使答案与证据之间形成显式绑定，进一步提升可解释性与可核验性。

第五步，可以加强回答端的工程稳健性。缓存、重试、降级策略与接口替代方案，都可以直接提升系统的连续可用性，并使整个问答链路更加稳定。

从这个角度看，当前原型最值得保留的是这条方法主线：**页面作为视觉检索对象、多向量页面重排、检索与生成明确分层**。沿着这条主线继续推进，系统可以自然扩展到更细的证据粒度、更强的多页组织能力与更稳健的回答端支持。
