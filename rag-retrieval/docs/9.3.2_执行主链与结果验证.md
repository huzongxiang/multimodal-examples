# 9.3.2 执行主链与结果验证

页面级文档问答是否真正成立，不能只看系统能不能启动，还要看页面索引、页面检索和多模态回答是否形成了清楚的因果链。当前样例的结果正好可以据此判断：页面级检索是否已经成立，生成阶段又在哪些地方暴露出明确边界。

## 1. 脚本入口与验证思路

仅仅说明系统“能运行”，并不能证明页面级检索与问答链路真正成立。要验证这一点，必须把输入条件固定下来，使同一文档、同一问题集和同一运行顺序可以被重复观察。为此，项目把整条主链收束为脚本 `run_pdf_qa.py`：它不依赖交互界面，而是直接执行页面索引、页面检索、多模态回答与结果保存。

运行方式如下：

```bash
cd <project_root>
micromamba run -n <env_name> python run_pdf_qa.py
```

这次案例运行选用的是一个两页证据较为集中的补充材料样例。脚本默认索引前 `2` 页，并围绕标题、会议信息、表格读取和图表面板识别四类问题进行验证。这样的设置并不追求题量规模，而是有意让不同证据形态都进入同一条问答主链。

默认问题集如下：

```python
DEFAULT_QUESTIONS = [
    "What is the title of this supplementary material?",
    "Which conference submission is this supplementary material for?",
    "What datasets are listed in Table 4?",
    "What four metric panels are shown in Figure 10?",
]
```

脚本还保留了一个非常有价值的分离式验证模式。如果只想验证本地页面级检索是否成立，而暂时不调用外部多模态生成接口，则可以使用：

```bash
micromamba run -n <env_name> python run_pdf_qa.py --retrieval-only
```

`retrieval-only` 模式的意义不在于少跑一步，而在于把“召回是否正确”与“回答是否稳定”拆开验证。这样一来，即便回答端出现限流或接口波动，也不会反过来掩盖检索链本身的质量。

## 2. 从页面索引到答案生成的执行闭环

从脚本实现看，整条主链可以概括为四个环节：建立一次独立运行、索引页面、检索相关页、再把命中页交给回答模型。代码如下。

```python
run_id = f"{pdf_path.stem}-{uuid.uuid4().hex[:8]}"
rag = None if args.retrieval_only else Rag()
middleware = Middleware(run_id, create_collection=True)
```

第一步的作用，是把每次运行放进独立上下文，避免不同实验之间的索引状态互相污染。第二步进入索引阶段，系统将 PDF 页面转换为图像并写入页面级表示：

```python
indexed_pages = middleware.index(
    pdf_path=str(pdf_path),
    id=run_id,
    max_pages=args.max_pages,
)
```

第三步，对每个问题执行页面级检索，确定后续回答所依据的证据页：

```python
search_result = middleware.search([question])[0]
top_score, top_doc_id = search_result[0]
page_num = top_doc_id + 1
image_path = BASE_DIR / "pages" / run_id / f"page_{page_num}.png"
```

第四步，在非 `retrieval_only` 模式下，将命中页图像交给 `GLM` 生成答案，并把本次观察结果写出：

```python
results.append(
    {
        "question": question,
        "answer": answer,
        "page_num": int(page_num),
        "search_score": float(top_score),
        "image_path": str(image_path.resolve()),
        "glm_attempts": int(attempts),
    }
)
```

这里最值得注意的一点是：在进入生成阶段之前，证据页已经被选定。也就是说，回答端失败与否，并不会改变检索端已经做出的页面判断。正因为有了这一层清晰分工，后面的结果分析才能真正分辨“问题出在检索”还是“问题出在生成”。

## 3. 本地页面级检索结果

在独立检索模式下，系统针对前三个问题只执行索引与页面召回，不调用外部回答模型。

表9.3.1 本地页面级检索结果

| 问题 | 命中页 | 检索得分 | 说明 |
| --- | --- | --- | --- |
| `What is the title of this supplementary material?` | 1 | `16.58` | 标题信息集中于首页页眉区域 |
| `Which conference submission is this supplementary material for?` | 1 | `13.00` | 会议信息同样位于首页顶部文本区域 |

这组三个结果有两个观察值得强调。第一，三类问题虽然问法不同，但都被稳定路由到第 `1` 页，说明问题表示已经能够与页面级视觉表示形成可比较的匹配关系。第二，标题、会议信息和表格读取并不依赖同一种证据形态，但当前系统仍然能把它们归并到同一页面检索链中。这意味着页面级检索并不是只对纯文本有效，而是已经能够覆盖文档中的多种视觉证据。

从系统分析角度看，这一步非常关键。因为在回答端尚未介入时，检索链已经给出了稳定的证据定位结果。后续如果多模态生成出现失败，就可以更明确地判断故障位于回答端，而不是本地页面索引与召回过程。

## 4. 全流程结果与案例分析

在完整问答模式下，系统对四个问题给出了如下结果。

表9.3.2 完整问答结果

| 问题类型 | 命中页 | 结果摘要 | 说明 |
| --- | --- | --- | --- |
| 标题识别 | 1 | 因接口限流未返回稳定答案 | 检索阶段正常 |
| 会议投稿信息 | 1 | `CVPR 2025` | 文本类问题成功 |
| `Table 4` 数据集提取 | 1 | `MVTec AD, VisA, BTAD, DAGM, DTD-Synthetic` | 表格类问题成功 |
| `Figure 10` 面板识别 | 2 | `1-shot AUROC, 1-shot PRO, 4-shot AUROC, 4-shot PRO` | 图表类问题成功 |

这一结果可以从三个层面理解。

第一，页面路由是成立的。前三个问题都集中命中第 `1` 页，而图表面板问题稳定命中第 `2` 页，说明系统并不是机械地把所有问题都送到同一页，而是能够根据问题内容在页面之间切换证据来源。对于一个只索引前两页的最小案例，这已经能够表明页面级检索主链在当前样例中是有效的。

第二，当前原型已经覆盖了文本、表格和图表三种不同证据形态。会议信息问题对应的是页眉文本，`Table 4` 问题对应的是表格单元格内容，`Figure 10` 问题对应的是图表面板标签。三类问题的成功表明，这个系统不是只会“看文字”，而是已经能够在整页图像上处理更复杂的文档视觉证据。

图9.3.2 页面级检索命中的第 2 页：Figure 10 所在页

![图9.3.2 页面级检索命中的第 2 页：Figure 10 所在页](../colpali-milvus-multimodal-rag-master/pages/test-95716e2e/page_2.png)

从图9.3.2可以看到，`Figure 10` 占据了第 `2` 页右上区域，四个子图标题分别对应 `1-shot AUROC`、`1-shot PRO`、`4-shot AUROC` 与 `4-shot PRO`。这一页与前一页的证据形态明显不同，但系统仍然把图表问题稳定路由到第 `2` 页，说明检索链并不是仅凭正文文字做匹配，而是已经能够对页面中的图表结构建立有效关联。

第三，当前链路的主要不稳定因素并不在本地检索，而在外部生成接口。标题问题的检索得分其实是四个问题中最高的一类，但回答端在连续五次重试后仍然返回 `429` 限流错误。这一现象恰好证明了把检索与生成拆开的价值：即便回答失败，系统仍然能够知道证据页已经被正确定位，因此不会把外部接口问题误判为检索问题。

因此，这一案例的结果不应被理解为“做了一个简单 demo”，而更合适地理解为“验证了一条页面级多模态文档问答主链”。这些结果支持以下判断：页面作为检索单元是可行的，本地多向量检索与多模态回答可以形成闭环，而实际运行中暴露出来的限制也具有明确的技术指向。
